SQL Injection (SQLi) - introduction and SQLi in login fields


After learning in the previous articles how to install the Linux terminal on PC and Android, how to use this OS, what are the main commands in Linux, as well as various legal aspects about hacking, in today's article it is time to move on to actual ethical hacking.

Note: Hacking a device or website without the owner's consent is illegal. Hacking techniques can only be used legally in environments specifically designed for testing such skills, on your own devices, or in a setting where you have the owner's permission.

The first technique we will talk about is called SQL Injection. By exploiting this type of vulnerability, an attacker can gain unauthorized access to a website's database and can execute commands that interact with the database. In this multi-part lesson, you will learn what SQL Injection is, how to legally exploit this vulnerability, and how to protect yourself against this type of attack.


How does SQL Injection works

First of all, we need to understand that website databases are created using a programming language called SQL (Structured Query Language). As you probably already know, each programming language has its specific usage: HTML is used for creating websites, CSS for styling the content, JavaScript for creating scripts, PHP for the connection between websites and servers etc. Therefore, SQL is used for creating databases and managing the connection between a website and its database.

Let's imagine that we have a website that has, in its database, the following table called "users", that contains the users's login informations:

ID Username Password City
1 john_doe password123 New York
2 jane_smith qwerty456 Los Angeles
3 mike_jones letmein789 Chicago
4 anna_white securepass New York
5 chris_brown admin123 Los Angeles

In the case of a login form, what the server sees is the following SQL query:

SELECT * FROM users WHERE username = ' ' AND password = ' '


Now, let's enter a valid set of credentials:

Username: john_doe
Password: password123

The SQL interrogation to the database in our case looks like this:

SELECT * FROM users WHERE username = 'john' AND password = 'password123'

After that, the server compares the entered credentials with those stored in the database. It checks if the combination john:password123 exists in the users table. Since the provided credentials are correct, the interrogation will return the TRUE value, granting access to the website.

If we enter incorrect credentials, such as mary:123456, the query will look like this:

SELECT * FROM users WHERE username = 'mary' AND password = '123456'

Since this combination does not exist in the database, the interrogation will return the FALSE value, and we will not be able to log in.

SQL Injection is based on the modification of the input fields, to manipulate the TRUE or FALSE values from the database interogation.

As we saw earlier, for a successful login, the query must return the TRUE value. But what if, instead of entering the correct credentials (which we may not know), we close the username field (whose name the server search in the database), and open another field where we enter a correct mathematical relation or another type of equality, making the server to choose the second value. Thus, our input will looks like this:

Username: mary' OR '1'='1
Password: 123456' OR '1'='1

Now, the interrogation is:

SELECT * FROM users WHERE username = 'mary' OR '1'='1' AND password = '123456' OR '1'='1'

Thus, the server checks whether the user "mary" exists or if 1=1. Since 1=1 is always TRUE, the result of the query for the username field will be TRUE. The same applies to the password field, where the server checks whether the password "123456" exists or if 1=1. Since the query result is TRUE, we will be able to log into the website. With such an input, the login will be performed in the account of the the first user in the database (usually the administrator).

There are several methods to exploit this vulnerability. In SQL, to mark a line of code that should be ignored by the programming language when executing the query (like a comment in human language created by the programmer that should not be considered by the program), we use -- (two minuses). Thus, everything that follows on that line after -- will be ignored by SQL (similar to the # in Python or PHP).

The user can enter the following input:

Username: a' OR '1='1' --
Password: 123456

In this case, the SQL query will look like this:

SELECT * FROM users WHERE username = 'a' OR '1'='1' --' AND password = 'doesntmatter'

The query for checking the password, being after the --, will be completely ignored. Thus, by using this tactic, the attacker can log in to the page of a user by only knowing the username, with the password no longer being necessary. An input like:

Username: john_doe' --
Password: doesntmatter

Will produce the following SQL interrogation:

SELECT * FROM users WHERE username = 'john_doe' --' AND password = ''

Practically speaking, it this case, the server only verify if the username is ok, then it let you in. Also, we can connect to a specific account based on its ID in the database. In general, the column containing the user ID is named "id". This column is automatically created by the database. Essentially, it represents the user's number in the order in which the account was created (the first user will be assigned ID 1, the second one ID 2, and so on).

Exploiting this, an attacker could try to enter the following input:

Username: a' OR id=2
Password: a' OR '1'='1

Or

Username: a' OR id=2 --
Password: doesntmatter

An input like this will log the attacker in the account of the second user in the database, in our case as jane_smith.
