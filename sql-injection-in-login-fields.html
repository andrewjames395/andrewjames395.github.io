SQL Injection (SQLi) - introduction and SQLi in login fields

After learning in the previous articles how to install the Linux terminal on PC and Android, what the main commands in Linux are, how to use this operating system, as well as various legal aspects about hacking, in today's article it is time to move on to actual ethical hacking.

Note: Hacking a server or website without the owner's consent is illegal. Hacking techniques can only be used legally in environments specifically designed for testing such skills, on your own devices, or in a setting where you have the owner's permission.


The first technique we will talk about is called SQL Injection. By exploiting this type of vulnerability, an attacker can gain unauthorized access to a website's database and can execute commands that interact with the database. In this multi-part lesson, you will learn what SQL Injection is, how to legally exploit this vulnerability, and how to protect yourself against this type of attack.


How does SQL Injection works

First of all, we need to understand that website databases are created using a programming language called SQL (Structured Query Language). As you probably already know, each programming language has its specific usage: HTML is used for creating websites, CSS for styling the content, JavaScript for creating scripts, PHP for the connection between websites and servers etc. Therefore, SQL is used for creating databases and managing the connection between a website and its database.

Let's imagine that we have a website that has, in its database, the following table called "users", that contains the users's login informations:

id username password city
1 john_doe password123 New York
2 jane_smith qwerty456 Los Angeles
3 mike_jones letmein789 Chicago
4 anna_white securepass New York
5 chris_brown admin123 Los Angeles

In the case of a login form, what the server sees is the following SQL query:

SELECT * FROM users WHERE username = ' ' AND password = ' '

The explanation of the query in human language is: Select all from the "users" table where the value of the "username" column is ' ' and the value of the "password" column is ' '.

Now, let's enter a valid set of credentials:

Username: john_doe Password: password123

The SQL interrogation to the database in our case looks like this:

SELECT * FROM users WHERE username = 'john' AND password = 'password123'

After that, the server compares the entered credentials with those stored in the database. It checks if the combination "john_doe:password123" exists in the users table. Since the provided credentials are correct, the interrogation will return the TRUE value, granting access to the website. Basically, the server asks the database: Does the username "john_doe" exists? TRUE! Does he has the password "password123"? TRUE! Then you can enter.

If we enter incorrect credentials, such as "mary:123456", the query will look like this:

SELECT * FROM users WHERE username = 'mary' AND password = '123456'

Since this combination does not exist in the database, the interrogation will return the FALSE value, and we will not be able to log in.

SQL Injection is based on the modification of the input fields, to manipulate the TRUE or FALSE values from the database interogation. As we saw earlier, for a successful login, the query must return the TRUE value. But what if, instead of entering the correct credentials (which we may not know), we close the username field (whose name the server search in the database), and open another field where we enter a correct mathematical relation or another type of equality, making the server to choose the second value. Thus, our input will looks like this:

Username: mary' OR '1'='1 Password: 123456' OR '1'='1

Now, the interrogation becomes:

SELECT * FROM users WHERE username = 'mary' OR '1'='1' AND password = '123456' OR '1'='1'

Thus, the server checks whether the user "mary" exists or if 1=1. Since 1=1 is always TRUE, the result of the query for the username field will be TRUE. The same applies to the password field, where the server checks whether the password is "123456" or if 1=1. Since the query result is TRUE, we will be able to log into the website. With such an input, the login will be performed in the account of the the first user in the database (usually the administrator).

There are several methods to exploit this vulnerability. In SQL, to mark a line of code that should be ignored by the programming language when executing the query (like a comment in human language created by the programmer that should not be considered by the program), we use -- (two minuses). Thus, everything that follows on that line after -- will be ignored by SQL (similar to the # in Python or PHP). The user can enter the following input:

Username: a' OR '1='1' -- Password: 123456

In this case, the SQL query will look like this:

SELECT * FROM users WHERE username = 'a' OR '1'='1' --' AND password = 'doesntmatter'

The query for checking the password, being after the --, will be completely ignored. Thus, by using this tactic, the attacker can log in to the page of a user by only knowing the username, with the password no longer being necessary. An input like:

Username: john_doe' -- Password: doesntmatter

Will produce the following SQL interrogation:

SELECT * FROM users WHERE username = 'john_doe' --' AND password = 'doesntmatter'

Practically speaking, it this case, the server only verify if the username is ok, then it let you in. Depending on how the interrogation is created, many times we have to use # instead of --, but we will discuss this in another article. Also, we can connect to a specific account based on its ID in the database. In general, the column containing the user ID is named "id". This column is automatically created by the database. Essentially, it represents the user's number in the order in which the account was created (the first user will be assigned ID 1, the second one ID 2, and so on).

Exploiting this, an attacker could try to enter the following input:

Username: a' OR id=2 -- Password: doesntmatter

An input like this will log the attacker in the account of the second user in the database, in our case jane_smith. We can continue with the testing, and instead of using "id," we can use other columns from the table. For this, we need to know the column names, although they can often be guessed using the registration form. We can input:

Username: a' OR city='Los Angeles' -- Password: doesntmatter

Using such credentials will log us into the account of the first user in the database who lives in Los Angeles,în our case jane_smith. Additionally, we can log in even if we only know the password but not the username by entering:

Username: a' OR '1'='1 Password: letmein789

There are countless combinations like this that we can use. Moreover, in the case of a website vulnerable to SQL Injection, an attacker can do much more than just log into different user accounts. They can execute commands in the database, such as adding new users, changing existing passwords, deleting users, or even wiping the entire table. In this scenario, SQL programming knowledge is an advantage. However, even without programming skills, this is not a problem, as there are tools that can automate the process.

It is also very important that, when legally exploiting an SQLi vulnerability, our input to be syntactically correct (e.g., avoiding misplaced apostrophes or incorrect syntax), to ensure a successful connection.


How do I test for SQL Injection

To check if a login field is vulnerable to SQL Injection, you can enter an apostrophe in one of the fields or use another character that interacts with SQL logic (such as ;, ', or "). If, after pressing the login button, you receive an output like "SQL Syntax Error", then the site is most likely vulnerable.


Where can I prectice SQLi attack legally

There are several websites on the internet where this technique can be legally practiced. One of them is altoro.testfire.net, which simulates a banking website and is intentionally designed to be vulnerable to SQL Injection.

Another is juice-shop.herokuapp.com, which simulates a juice shop and is intentionally built with multiple types of vulnerabilities, including SQL Injection in the login fields.


How common is the SQLi vulnerability in login fields today?

Even if SQL injection is one of the most common vulnerability, however, in the current year, this type of SQLi (in login field injection) is very difficult to find. It may still be encountered on older websites that have not been maintained or maybe on websites from less developed countries, but human error can occur anywhere.
